# 代理配置

在某些地区（如中国大陆），GitHub API 的访问可能不稳定。Gleaner 支持配置代理服务来改善连接质量。

## 工作原理

配置代理后，所有 GitHub API 请求的 URL 会被加上代理前缀：

```
原始请求: https://api.github.com/repos/owner/repo/git/trees/main
代理请求: https://gh-proxy.com/https://api.github.com/repos/owner/repo/git/trees/main
```

同样，图片和视频等媒体资源也会通过代理下载。

## 配置方法

在设置页面的「访问令牌」选项卡中，找到「GitHub API 代理」输入框：

1. 输入代理地址，如 `https://gh-proxy.com`
2. 点击保存令牌
3. 代理立即生效，后续所有请求都会通过代理

> 留空则直连 GitHub。

## 代理要求

代理服务需要满足以下条件：

- 支持 **CORS**（`Access-Control-Allow-Origin: *`）— 因为 Gleaner 是浏览器端应用
- 支持 **转发完整 URL** — 代理格式为 `{proxy}/{原始URL}`
- 支持 **转发请求头** — 包括 Authorization 头（用于 [[GitHub Token 配置|Token]] 认证）

## 推荐代理

| 代理 | 地址 | 说明 |
|------|------|------|
| gh-proxy.com | `https://gh-proxy.com` | 支持 CORS，兼容 API 请求 |

> 注意：很多 GitHub 加速服务（如 ghproxy.vip）只支持 `git clone` 和 Release 下载，不支持 REST API 代理或缺少 CORS 头，不适用于 Gleaner。

## 媒体代理

代理不仅用于 API 请求，也会自动处理：

- **图片** — Markdown 中引用的仓库内图片
- **视频** — `<video>` 标签中的 GitHub 链接会自动转换为 `raw.githubusercontent.com` 格式后通过代理下载

## 验证代理

配置后，可以通过以下方式验证代理是否生效：

1. 在设置中添加一个仓库并同步
2. 如果同步成功且文件正常加载，代理配置正确
3. 如果出现网络错误，检查代理地址是否正确、是否支持 CORS
