# 技术架构

本文介绍 Gleaner 的内部工作原理，适合希望深入了解或参与贡献的开发者。

## 整体架构

Gleaner 是一个纯前端单页应用（SPA），没有后端服务。所有数据通过 GitHub REST API 获取，缓存在浏览器的 IndexedDB 中。

```
┌─────────────────────────────────────────────────┐
│                   浏览器                          │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │
│  │ React UI │←→│  Zustand  │←→│   IndexedDB  │   │
│  │          │  │  Store    │  │   (Dexie.js) │   │
│  └──────────┘  └──────────┘  └──────────────┘   │
│       ↕                            ↕              │
│  ┌──────────┐              ┌──────────────┐      │
│  │ 路由系统  │              │  Sync Engine │      │
│  │(React    │              │  (后台同步)   │      │
│  │ Router)  │              └──────┬───────┘      │
│  └──────────┘                     │               │
│                                   ↓               │
│                          GitHub REST API           │
│                          (可选: 通过代理)          │
└─────────────────────────────────────────────────┘
```

## 技术栈

| 层面 | 技术 |
|------|------|
| 框架 | Vite + React 19 + TypeScript |
| 状态管理 | Zustand |
| 本地数据库 | IndexedDB via Dexie.js |
| 样式 | Tailwind CSS 4 + shadcn/ui |
| Markdown | unified (remark → rehype) |
| 图谱 | react-force-graph-2d |
| 测试 | Playwright (E2E) |

## 数据模型

IndexedDB 中有以下主要表：

### repos
存储仓库元信息：同步状态、tree SHA、文件计数等。tree SHA 用于增量同步判断。

### files
存储 Markdown 文件：路径、内容、标题、SHA。SHA 用于检测文件变更，避免重复下载。

### links
存储解析后的链接关系：源文件、目标标题、目标文件 ID。支持 [[双向链接|wikilinks]] 和标准 Markdown 链接。

### profiles
存储 [[配置文件管理|Profile]] 配置：类型、YAML 内容或 GitHub 配置仓库地址。

## 同步流程

```
1. 获取仓库 Git Tree（一次 API 调用）
2. 对比 tree SHA — 如果相同则跳过
3. 筛选 .md 文件，应用路径过滤
4. 对比每个文件的 SHA — 只下载变更的文件
5. 逐个下载文件内容（每个文件一次 API 调用）
6. 解析 wikilinks 和 Markdown 链接
7. 构建链接索引
```

### 增量同步

Gleaner 通过两级 SHA 对比实现增量同步：

- **Tree SHA** — 判断仓库是否有任何变更
- **File SHA** — 判断具体哪些文件发生了变更

这意味着第二次同步通常只需 1 次 API 调用（获取 tree），如果没有变更则立即完成。

### 速率限制处理

当遇到 GitHub API 速率限制（403/429）时，同步引擎会：

1. 暂停当前文件的下载
2. 等待到速率限制重置时间
3. 自动重试，不跳过任何文件

详见 [[GitHub Token 配置]] 了解如何提升限额。

## Markdown 渲染管线

```
原始 Markdown
    ↓ remark-parse (解析为 AST)
    ↓ remark-gfm (GitHub Flavored Markdown)
    ↓ remark-rehype (转换为 HTML AST)
    ↓ rehype-raw (处理内嵌 HTML)
    ↓ rehype-slug (标题锚点)
    ↓ rehype-highlight (代码高亮)
    ↓ rehype-react (渲染为 React 组件)
渲染结果
```

自定义组件处理：
- `img` → RepoImage（仓库内图片通过 API 获取，支持 [[代理配置|代理]]）
- `video` → RepoVideo（自动转换 GitHub URL 为 raw 地址）
- `a` → MdLink（处理 wikilinks 和内部链接的点击导航）
- `pre` → CodeBlock（代码复制按钮）
- `table` → ResponsiveTable（响应式表格）

## 链接解析

[[双向链接|Wikilinks]] 的解析分两步：

1. **提取** — 从 Markdown 内容中正则匹配 `[[target]]` 和 `[[target|alias]]`
2. **解析** — 在所有已缓存文件中查找匹配的标题或文件名，当前仓库优先

解析结果存储在 `links` 表中，供反向链接面板和 [[知识图谱]] 使用。
